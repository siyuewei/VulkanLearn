# Vulkan基础概念

学习Vulkan的时候觉得概念太多太杂了，需要自定义的东西非常多。下面做一个框架的梳理。

## 一、基建部门 (Initialization)

这部分代码写一次基本就不动了，属于公司注册和厂房搭建阶段。

### VulkanInstance

**公司营业执照**。
初始化 Vulkan 的第一步，告诉驱动程序你是谁，你要用什么扩展功能（比如是否开启调试层）。没有这个实例，后续一切免谈。

### VulkanSurface

**展示橱窗**。
Vulkan 本身是个纯粹的计算和绘图 API，它不负责和 Windows/Linux 系统打交道。Surface 就是连接 Vulkan 内核与桌面窗口系统的桥梁。

### VulkanPhysicalDevice

**选购设备**。
电脑上可能集显和独显都有。这一步是遍历硬件，挑出那个最适合干活的显卡（通常选独显）。

### VulkanLogicalDevice

**雇佣操作员**。
选中显卡后，我们需要创建一个逻辑上的接口来操作它。在这里开启我们需要用到的特性（比如几何着色器、各向异性过滤等）。

### VulkanQueue & VulkanQueue Family

**传送带**。
显卡不听口头指挥，只认传送带。我们把命令提交到 Queue 上，显卡会排队执行。一般有图形队列（画画）和呈现队列（显示）两种。

**Queue Family**就是这些Queue的组合，一个Queue Family里面有很多个相同的Queue，类似由很多相同流水线组成的车间。显卡内部有不同的电路负责不同的事情：

- Graphics Queue Family (图形部门)：擅长画画（画三角形、光栅化）。
- Compute Queue Family (计算部门)：擅长纯算数（科学计算、AI 推理，不涉及画画）。
- Transfer Queue Family (物流部门)：擅长搬运数据（内存拷贝），画画能力为 0。

有的显卡（如高端 N 卡）：有一个全能部门（Graphics + Compute + Transfer 都在一个 Family 里）。

有的显卡：分得很细，画画的只能画画，搬运的只能搬运。

在初始化时遍历 Queue Family，其实是在问显卡：“请问哪一个部门（Family）懂画画？把它找出来，我以后把画画的任务发给这个部门下面的传送带（Queue）。”

## 二、画布供应部门 (SwapChain)

基建搞定后，需要准备画纸，并确立“画完怎么展示”的机制。

### VulkanSwapChain

**轮播画架**。
屏幕刷新率通常是 60Hz。如果我们直接在屏幕显示的这张纸上画，画到一半屏幕刷新了，就会产生画面撕裂。
SwapChain 建立了一个缓冲池（通常 2-3 张图）。显卡在后台画好一张，直接把画架转过来给屏幕看，这就是“交换”的概念。

### VulkanImage & VulkanImageView

**裸纸与相框**。

- **Image**：是显存里真正的像素数据，但它只是一堆 RAW 数据。
- **ImageView**：Vulkan 比较死板，不能直接用 Image，必须给它加个“相框”或者“说明书”，告诉管线：“这是一张 2D 的 RGB 图片，请按这个格式去读”。

### VulkanFrameBuffer

**综合画板**。
它是一个容器，把具体的 SwapChain（哪一张图像）与 RenderPass（渲染流程）绑定在一起。它明确了：“这一帧，我到底要画在哪几张特定的纸上”。

## 三、绘画规则部门 (Pipeline)

这是最复杂的部分，Vulkan 要求在动笔前把所有规则定死。

### Shader (着色器)

**绘画技巧**。

- Vertex Shader：告诉显卡点在哪里（定形状）。
- Fragment Shader：告诉显卡颜色是什么（上色）。

### RenderPass (渲染流程)

**施工蓝图**。
它不关心具体画什么物体，它关心的是宏观流程：

- 开始画之前要不要清空画布？（比如涂黑）。
- 画完之后画纸要去哪？（是存起来给下一帧用，还是直接送去显示）。
- 多张图（颜色图、深度图）怎么配合？

### Pipeline (管线)

**全自动流水线机器**。
这是大总管。它把 Shader、视口大小、线框模式、深度测试开关等所有配置熔铸成一个不可修改的对象。（后文详细解释）。

## 四、数据中心 (Buffers)

这里存放绘画所需的原材料。

### VertexBuffer (顶点缓冲)

**模型仓库**。
存放物体的几何数据，比如三角形的三个角坐标、颜色、纹理坐标。

### IndexBuffer (索引缓冲)

**目录索引**。
为了省空间，顶点可以复用。这里存放“0-1-2”这样的连接序号，告诉显卡按什么顺序把点连成线。

### UniformBuffer (统一缓冲)

**全局参数**。
存放那种每一帧都在变的少量数据，比如 MVP 变换矩阵（控制物体旋转、缩放、移动）、光照参数等。显卡里的 Shader 就像一个函数，UBO 就是传给它的“全局变量”。

### Descriptor Set Layout (描述符布局)

**接口/插座设计图 (模具)**。
Pipeline（管线）虽然是写死的机器，但它预留了一些**接口**用来接数据。Layout 就是告诉管线：“这里有一个三孔插座（UBO），那里有一个两孔插座（纹理）”。只有定义了 Layout，管线才知道去哪里读数据。它是**只读的模板**。

- **VkDescriptorSetLayoutBinding (单个插座描述)**：
  描述**某一种特定插座**的规格。
- `descriptorType`：插座类型（是插 UBO 的还是插纹理的？）。
- `descriptorCount`：**排插数量**。如果设为 1，就是一个单孔插座；如果设为 10，说明这是一个有 10 个孔的排插，但每个孔插的东西类型都一样（比如 `uniform vec4 lights[10]`）。
- **VkDescriptorSetLayoutCreateInfo (整体施工图)**：
  描述**整面墙上所有的插座**。
- `pBindings`：把上面定义好的各种 Binding 列个表传进去。
- `bindingCount`：**插座种类数**。比如墙上既有 UBO 插座，又有纹理插座，那这里就是 2。它统计的是你有多少个不同的 `Binding` 定义。

### VkDescriptorSet (描述符集)

**真实的插排/具体的卡片 (实例)**。
Layout 只是图纸，不占显存，也不能存数据。Set 才是真正按照图纸造出来的**实物**。

- **区别**：
- **Layout (模具)**：规定了卡上有“姓名”、“照片”两个框。（全公司只有一种工牌模板）。
- **Set (实例)**：是具体的卡片，里面填了“张三”、“张三的照片”。（每个人都有一张不同的卡，但结构都一样）。

- **用法**：

1. 拿着 Layout (图纸) 向显卡申请创建一个 Set (实物)。
2. 把真正的 Buffer (数据) 或 Image (图片) **更新 (Update)** 到 Set 里面。
3. 在画画时，把这个 Set **绑定 (Bind)** 到管线上。

### StagingBuffer (中转缓冲)

**搬运车**。
显卡最核心的高速显存（Device Local）CPU 进不去。所以数据流通常是：CPU -> StagingBuffer (中转) -> GPU 内部指令搬运 -> VertexBuffer (高速显存)。

## 五、调度中心 (Command & Sync)

负责录制命令，并防止 CPU 和 GPU 打架。

### CommandPool & CommandBuffer

**指令池与指令本**。
Vulkan 的操作不是立即执行的，而是先录制在 CommandBuffer 上，然后一次性提交。Pool 是用来分配管理这些内存的。

### Semaphore (信号量)

**GPU 内部红绿灯**。
CPU 看不到它。这是显卡内部流水线之间的暗号。例如：“画画队列”告诉“显示队列”：我画完了，你可以拿去显示了。

### Fence (栅栏)

**CPU/GPU 对讲机**。
防止 CPU 跑太快把 GPU 撑死。CPU 每帧提交完任务就在栅栏这等着，直到 GPU 发信号说“我这一帧搞定了”，CPU 才会放行下一帧的逻辑。

# 补充与答疑

### Q1: 为什么 Command 需要一个 Command Pool？Pool 到底是什么概念？

**通俗解释：**
想象你在拍电影，Command Buffer 是“胶卷”，而录制的指令就是胶卷上的画面。
Command Pool 就是“胶卷仓库”。

**技术原因：**

1. **内存管理**：在底层，频繁地向操作系统申请和释放零碎的内存（创建/销毁 CommandBuffer）是非常昂贵且低效的。Pool 会一次性申请一大块内存，然后当你需要 CommandBuffer 时，直接从这个池子里切一块给你。
2. **线程绑定**：Vulkan 中，Pool 是和特定的 Queue Family（队列族）绑定的。图形队列有图形队列的池子，计算队列有计算队列的池子。这保证了多线程录制命令时互不干扰，无需加锁，性能极高。

### Q2: Pipeline 到底是个什么概念？它规划了多少内容？

**通俗解释：**
如果 OpenGL 是一个有很多开关的**“控制台”**（你可以随时按下开关，改变状态），那么 Vulkan 的 Pipeline 就是一个**“模具”**或者**“铸造好的机器”**。

**它规划了多少内容？**
几乎所有和“怎么画”有关的细节都被**焊死**在 Pipeline 里了。一旦创建，完全不可修改（Immutable）。如果你想换一种画法（比如从实心变成线框模式），你必须切换到另一个 Pipeline。

它具体包含（但不限于）以下内容，可以说是事无巨细：

1. **Shader 阶段**：用哪个顶点着色器，用哪个片段着色器。
2. **顶点输入**：数据结构长什么样（比如 `vec2 pos`, `vec3 color`）。
3. **输入装配**：画点、画线还是画三角形？
4. **视口与剪裁**：画在屏幕的哪个区域？（这部分也就是 Dynamic State 唯一允许微调的地方）。
5. **光栅化器**：是画实心还是线框？要不要剔除背面？线条多宽？
6. **多重采样**：抗锯齿怎么搞？
7. **深度与模板测试**：近处的物体遮挡远处的物体，规则是什么？
8. **颜色混合**：画半透明物体时，新颜色和旧颜色怎么叠加？
9. **Pipeline Layout**：在这个管线上留了几个“插座”（Uniform 接口）给外部传数据？

**总结**：Pipeline 就是一份及其详尽的**生产标准书**。显卡拿到这个对象，不需要做任何检查，直接全速运行，这就是 Vulkan 高效的原因。
